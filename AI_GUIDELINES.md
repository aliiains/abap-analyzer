# ABAP Code Analyser — Правила доработки через ИИ

Этот документ описывает соглашения, которым должен следовать ИИ-ассистент
(Claude, GPT и др.) при доработке проекта. Цель — сохранить качество кода,
архитектурную целостность и полноту тестового покрытия.

---

## 1. Допустимые изменения

### Низкий риск — можно вносить напрямую
- Добавление новой метрики (подкласс `BaseMetric` в `metrics.py`)
- Добавление нового детектора замечаний (подкласс `IssueDetector` в `report.py`)
- Расширение списка ключевых слов ABAP (`_ABAP_KEYWORDS` в `gui.py`)
- Изменение порогов метрик (значения `threshold()` в классах метрик)
- Добавление новых пунктов меню и кнопок в панель инструментов
- Исправление текстов сообщений, подписей, статусной строки
- Добавление новых тестов в существующие тест-классы
- Изменение цветовой схемы (`_PALETTE`, `_HEATMAP_COLOURS`)

### Средний риск — требует проверки тестами
- Изменение структуры `ParseResult`, `ABAPLine`, `Token` (влияет на все модули)
- Изменение структуры `MetricsReport` или `MetricValue`
- Изменение алгоритма подсчёта глубины вложенности (`ABAPParser`)
- Добавление новой вкладки в правую панель GUI
- Изменение формата текстового отчёта
- Рефакторинг внутри существующего метода без изменения его сигнатуры

### Высокий риск — требует особого внимания
- Изменение публичного интерфейса абстрактных классов (`BaseMetric`, `IssueDetector`, `BaseReportFormatter`)
- Изменение алгоритма лексера (`ABAPLexer.tokenize`)
- Замена структуры хранения данных в `ParseResult`
- Изменение порядка вызовов в `App._analyse()`
- Любое изменение, затрагивающее более двух модулей одновременно

## 2. Зоны повышенного риска

### `parser.py` — алгоритм вложенности
```
ВНИМАНИЕ: счётчик _depth должен:
- увеличиваться ПОСЛЕ фиксации глубины текущей строки для открывателей
- уменьшаться ДО фиксации глубины для закрывателей
Нарушение этого порядка ломает тепловую карту и метрику MaxNestingDepth.
```
**Признак поломки:** тест `test_depth_if_endif` падает.

---

### `metrics.py` — пороговые значения
```
ВНИМАНИЕ: уровни ok/warning/error жёстко используются в report.py
для классификации замечаний. Изменение строк "ok", "warning", "error"
потребует правки report.py и тестов.
```

---

### `gui.py` — порядок слоёв тегов Tkinter
```
ВНИМАНИЕ: тепловая карта (фоновые теги) должна применяться ДО
синтаксической подсветки (теги foreground). Иначе ключевые слова
теряют цвет. Порядок в _apply_heatmap():
  1. HeatmapPainter.paint()
  2. SyntaxHighlighter.highlight()
```
**Признак поломки:** ключевые слова ABAP не окрашиваются при включённой тепловой карте.

---

### `gui.py` — метод `_paste()`
```
ВНИМАНИЕ: _paste() должен возвращать "break", иначе Tkinter выполнит
вставку дважды (стандартный обработчик + наш). Не убирайте return "break".
```

---

### Зависимости между модулями
```
gui.py  →  parser.py  →  (нет зависимостей)
gui.py  →  metrics.py →  parser.py
gui.py  →  report.py  →  parser.py, metrics.py
```
`parser.py`, `metrics.py`, `report.py` **не импортируют** `gui.py` и `main.py`.
Нарушение этого правила создаёт циклические импорты.

## 3. Обязательные проверки после изменений

### После любого изменения
```bash
# Запустить все тесты
python -m pytest tests/ -v

# Ожидаемый результат: 169 passed (или больше, если добавлены новые тесты)
```

### После изменения `parser.py`
```bash
# Обязательно запустить полный набор тестов парсера
python -m pytest tests/test_parser.py -v

# Проверить глубину вложенности вручную на примере:
# IF x. LOOP AT t. ENDIF. ENDLOOP. — должно: IF=1, LOOP=2, строка после ENDIF=1, после ENDLOOP=0
```

### После изменения `metrics.py`
```bash
python -m pytest tests/test_metrics.py -v

# Проверить что все 10 метрик присутствуют в отчёте:
python -m pytest tests/test_metrics.py::TestMetricsCalculator -v
```

### После изменения `report.py`
```bash
python -m pytest tests/test_report.py -v
```

### После изменения `gui.py`
```bash
# Автотестов нет — проверить вручную:
python main.py
# 1. Вставить пример кода (Ctrl+V или «Загрузить пример»)
# 2. Нажать «Анализировать»
# 3. Проверить подсветку, тепловую карту, отчёт
# 4. Переключить «Подсветка» и «Тепловая карта» — кнопки должны менять метку
# 5. Нажать «Очистить» — оба поля должны очиститься
```

### После добавления новой зависимости (pip-пакет)
```bash
# Убедиться что приложение запускается без ошибок
python main.py

# Добавить зависимость в requirements.txt если это runtime-зависимость
# (не нужно для стандартной библиотеки Python)
```

### Перед сборкой .exe
```bash
# Все тесты должны пройти
python -m pytest tests/ -v

# Затем сборка
python build_exe.py
```

## 4. Требования к новому коду

### Стиль
- Все публичные методы и классы — docstring на русском или английском
- Аннотации типов (`-> None`, `-> str`, `list[int]`) обязательны для публичных методов
- Строки не длиннее 100 символов
- Никаких `print()` в production-коде — только GUI-статусная строка или логирование

### Архитектурные правила
- Новая метрика = новый класс, наследник `BaseMetric`, в `metrics.py`
- Новый детектор = новый класс, наследник `IssueDetector`, в `report.py`
- Бизнес-логика не живёт в `gui.py` — только вызовы существующих компонентов
- `gui.py` не импортируется ни одним другим модулем

### Тесты
- Каждая новая метрика → минимум 5 тестов:
  - пустой ввод
  - минимальное значение (уровень `ok`)
  - значение на границе `warning`
  - значение в зоне `error`
  - типичный реальный сценарий
- Каждый новый детектор → минимум 3 теста:
  - случай когда замечание НЕ должно возникать
  - случай когда замечание ДОЛЖНО возникать
  - граничный случай (ровно на пороге)
- Тесты не должны зависеть от порядка выполнения

## 6. Что НЕ делать

| Запрещено | Почему |
|-----------|--------|
| Импортировать `gui.py` из `parser.py` / `metrics.py` / `report.py` | Циклический импорт |
| Добавлять бизнес-логику прямо в обработчики кнопок `App` | Нарушение SRP, не покрывается тестами |
| Убирать `return "break"` из `_paste()` | Двойная вставка текста |
| Менять порядок `paint()` → `highlight()` в `_apply_heatmap()` | Потеря цвета ключевых слов |
| Использовать `tk.Text.insert()` без учёта состояния `state` | Ошибка при попытке писать в `DISABLED`-виджет |
| Удалять тесты вместо того чтобы их починить | Скрывает реальные ошибки |
| Делать `_depth` отрицательным в парсере | Тепловая карта рисует неверные цвета |
